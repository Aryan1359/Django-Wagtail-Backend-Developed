Install & Setup Sequence Guide
Django + Wagtail + React + Bootstrap Project

Step-by-step guide with detailed explanations, reasoning, and use cases for each technology in your stack.
(Tip: See Glossary of Key Terms at the bottom if you’re new to any technical terms!)

1. Environment Preparation

Reason: A solid Python and Node environment ensures all future dependencies install cleanly and match between your local machine and production.
Use Case: Ensures your Python, Node.js, and system libraries are isolated and reproducible (via Anaconda) for both development and deployment.
  • Install Anaconda (if not already).
  • Create & activate your Conda environment:
    conda env create -f environment.yml
  • Install Python 3.11+ and Node.js (LTS) within this environment.
  • Test with: python --version and node --version
Note: Anaconda helps avoid “works on my machine” problems.

2. Initialize Version Control (Git)

Reason: Git tracks all changes, lets you roll back mistakes, collaborate safely, and integrate with GitHub, GitLab, etc.
Use Case: Required for safe, modern software development and easy cloud deployment later.
  • Initialize git: git init
  • Add a .gitignore (Python, Node, env files, etc.)
  • First commit: git add . && git commit -m "Initial commit"
Tip: Commit after each major step below!

3. Django + Wagtail Backend

Reason: Django is the main engine; Wagtail sits on top of it. Get your backend working before adding a frontend.
Use Case: Provides the database, admin panel, and API foundation for your whole project.
  • Install Django: pip install django
  • Start your Django project: django-admin startproject backend
  • Install Wagtail: pip install wagtail
  • Set up Wagtail: wagtail start mysite
  • Install PostgreSQL client: pip install psycopg2-binary or pg8000
  • Edit settings.py to use PostgreSQL (not SQLite for production!)
  • Migrate: python manage.py migrate
  • Create admin user: python manage.py createsuperuser
Why now? Setting up models, admin, and CMS features is easier before wiring in the API or frontend.

4. React + Bootstrap Frontend

Reason: Keeping the React app in its own folder separates backend logic from frontend UI, avoids confusion, and allows independent dev/reloads.
Use Case: Enables a modern, dynamic user experience and easier future expansion (mobile, etc).
  • Create React app: npx create-react-app frontend or npm create vite@latest frontend -- --template react
  • Install Bootstrap: npm install bootstrap
  • Set up Navbar, Footer, etc., as reusable components.
  • Use Axios or Fetch for API calls.
Tip: Develop the UI in parallel with the backend, connecting via APIs as soon as available.

5. API Layer (DRF & CORS)

Reason: The API bridges your Django backend (data, auth, CMS) with your React frontend. CORS enables secure cross-origin requests.
Use Case: Needed for React to fetch content, post forms, or do authentication with Django.
  • Install Django REST Framework: pip install djangorestframework
  • Add 'rest_framework' to INSTALLED_APPS
  • Install and set up CORS headers: pip install django-cors-headers
  • Add CORS settings to allow React frontend requests.
  • Build your first API endpoints (e.g., /api/pages/).
Why here? Once backend & frontend skeletons are up, APIs let you join them.

6. Wagtail CMS Extensions

Reason: Extensions like StreamField, Image Gallery, Forms, and custom page types turn basic Wagtail into a powerful CMS.
Use Case: Lets editors manage pages, images, galleries, and forms without coding; essential for any real site.
  • Add StreamField to page models for rich, flexible content.
  • Add wagtail.contrib.forms for contact forms.
  • Use wagtail.contrib.modeladmin for admin UI extensions.
  • Add wagtail.contrib.settings for site-wide settings.
  • Build custom Home, Blog, About page types, etc.
  • Set up the image gallery using Wagtail’s image chooser blocks.
Why now? Your CMS is ready for content, media, and custom structure before you fine-tune the user experience or deploy.

7. User Authentication

Reason: Secure login for staff, editors, or site users, plus optional support for React-based authentication (token/JWT).
Use Case: Lets you restrict pages, manage accounts, or offer protected APIs for users and admins.
  • Use Django’s built-in auth for admin and staff logins.
  • If React frontend needs user auth: Add JWT/Session support (django-rest-framework-simplejwt or similar).
  • Wire up login/logout/register APIs for frontend if required.
Tip: Don’t add JWT unless you really need true headless user login via React.

8. Celery & Redis (Async Tasks)

Reason: Needed if your project sends emails, processes images, or runs other background tasks that shouldn’t slow down page loads.
Use Case: Handle heavy/slow work in the background—e.g., contact form emails, video processing, notifications, etc. (See Celery and Redis in glossary).
  • Install Celery: pip install celery
  • Install Redis server (local or cloud), plus pip install redis
  • Configure Celery in Django (celery.py)
  • Test with a sample async task.
Optional: Only set this up if you know you’ll need background processing. Not required for static or simple sites.

9. Docker & DevOps

Reason: Containerization means your app runs the same everywhere—dev, staging, or prod. Also enables easy scaling and cloud deployment.
Use Case: Portable local dev, easy onboarding for team, seamless deploy to AWS, Heroku, etc.
  • Write Dockerfile for Django backend.
  • Write Dockerfile for React frontend (optional, if deploying separately).
  • Create docker-compose.yml to orchestrate backend, frontend, db, redis, etc. locally.
  • Add .env.example for environment variables.
  • Test local Docker build and up: docker-compose up --build
Why now? Prepares you for cloud deploy and keeps your environments consistent from local dev through to production.

10. Cloud Hosting & Deployment

Reason: Cloud hosting gets your app live, lets you collaborate, and test the full workflow.
Use Case: Run your site on AWS, Heroku, Render, DigitalOcean, or another cloud—serving real users or clients.
  • Prepare deploy scripts (for AWS ECS, Heroku, Render, etc.)
  • Set up production settings (static/media handling, allowed hosts, secure env vars, etc.)
  • Deploy with Docker or platform-native methods.
  • Test and monitor!
Tip: Deploy and test in the cloud before you “finish” the project—never wait until the end.

What is it? A browser security feature that restricts how web pages can request resources (like data) from a different server (domain) than the one the page came from.

Use Case: Django's built-in authentication system primarily uses session-based authentication for its admin panel. While suitable for server-rendered applications, it requires careful handling (like CSRF tokens) when used with a separate React frontend.
